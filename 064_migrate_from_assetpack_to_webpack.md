# Implementation Plan - Migrate from AssetPack to Webpack

This plan describes the migration of the openQA asset pipeline from the Perl-based `Mojolicious::Plugin::AssetPack` to `Webpack`. This migration will modernize the frontend build process, improve performance, and align with industry standards, while maintaining compatibility with the existing legacy codebase's execution model.

## User Story

As an openQA developer, I want to use modern frontend tooling (Webpack, Sass, etc.) so that I can develop and build the web UI more efficiently and leverage the modern JavaScript ecosystem, without breaking existing synchronous JavaScript assumptions in the templates.

## Current State (AssetPack)

- **Configuration**: Assets are defined in `assets/assetpack.def` (54+ bundles) and `assets/assetpack.yml`.
- **Processing**: Uses Perl modules (`CSS::Sass`, `JavaScript::Minifier::XS`) via AssetPack pipes.
- **Templates**: Assets are included using `%= asset 'bundle_name.ext'` and `icon_url 'name.png'`.
- **Build**: `tools/generate-packed-assets` triggers pre-generation during build/install.
- **Serving**: Assets are served from `public/asset/` with content-based checksums.

## Why Not Vite?

An initial attempt was made to migrate to Vite (see `tasks/049_migrate_from_assetpack_to_vite_8.md`). However, Vite fundamentally relies on native ES modules (`<script type="module">`) during development and (by default) in production.

This caused critical issues with the legacy openQA codebase:

- **ES Module Deferral**: ES modules are deferred by default. Many legacy openQA templates rely on inline scripts (e.g., `$(function() {...})` or immediate calls to globals) that execute _before_ the deferred ES modules have loaded, leading to `$` or other globals being undefined.
- **Test Coupling**: Selenium tests in the test suite (like `wait_for_ajax`) are tightly coupled to the immediate JavaScript execution context and immediate global availability.
- **Brittle Workarounds**: Attempts to expose globals (like jQuery) to the window object manually and implement readiness signaling mechanisms proved too brittle and invasive for the legacy templates.

## Proposed Architecture (Webpack)

- **Bundler**: Webpack.
- **Execution Model**: Webpack bundles multiple entry points with code-splitting (`optimization.splitChunks`) but outputs classic, synchronous scripts (IIFE/UMD) by default. This matches AssetPack's execution timing exactly and avoids the ES module deferral issues.
- **Integration**:
  - New Mojolicious plugin `OpenQA::Plugin::Webpack` to replace `AssetPack`.
  - **Development**: Proxies asset requests to the `webpack-dev-server` (HMR enabled).
  - **Production**: Uses `manifest.json` (generated by `webpack-manifest-plugin`) to map logical names to hashed production files in `public/dist/`.
- **Asset Organization**:
  - Entry points will be created in `assets/entry/` for each bundle currently in `assetpack.def`.
  - Sass/SCSS will be processed by `sass-loader` and extracted with `mini-css-extract-plugin`.

## Implementation Steps

### 1. Environment & Dependencies

- Update `package.json`:
  - Add `webpack`, `webpack-cli`, `webpack-dev-server`, `babel-loader`, `sass-loader`, `css-loader`, `style-loader`, `mini-css-extract-plugin`, `webpack-manifest-plugin`, `sass`, `postcss`, `autoprefixer` as dev dependencies.
  - Add scripts: `"dev": "webpack serve --mode development"`, `"build": "webpack --mode production"`.
- Update `dependencies.yaml` and `cpanfile` to mark `Mojolicious::Plugin::AssetPack` and related Perl modules for removal.

### 2. Webpack Configuration

- Create `webpack.config.js`:
  - Define all 54+ entry points from `assetpack.def` in the `entry` object.
  - Configure `output.path` to `../public/dist` and `output.publicPath`.
  - Configure `module.rules` for JS (babel-loader) and SCSS/CSS (mini-css-extract-plugin, css-loader, sass-loader).
  - Add `MiniCssExtractPlugin` and `WebpackManifestPlugin` to `plugins`.
  - Configure `optimization.splitChunks` for efficient code-splitting (e.g., separating vendor chunks).
  - Configure `devServer` for local development proxying.

### 3. Asset Restructuring & Entry Points

- Create a script (or manual process) to convert `assetpack.def` sections into Webpack entry points.
- Example `assets/entry/bootstrap.scss`:
  ```scss
  @import 'bootstrap/scss/bootstrap';
  @import '../stylesheets/openqa';
  // ... other imports from assetpack.def
  ```
- Example `assets/entry/bootstrap.js`:
  ```javascript
  import 'jquery';
  import '../javascripts/openqa';
  import 'bootstrap';
  // ... other imports from assetpack.def
  ```

### 4. Mojolicious Integration

- Implement `lib/OpenQA/Plugin/Webpack.pm`:
  - Add `webpack` helper (to replace or supplement `asset`).
  - Logic to handle `manifest.json` in production.
  - **Robustness**: Ensure the manifest is found relative to the project root, even when `OPENQA_BASEDIR` is set.
  - **Mapping**: Implement robust mapping from logical bundle names to Webpack manifest keys.
- Update `lib/OpenQA/Assets.pm` to register the new plugin and remove AssetPack.
- Update `lib/OpenQA/WebAPI/Plugin/Helpers.pm` to use the new system for `icon_url`.

### 5. Verification & Bug Fixes

- **Template Migration**: Ensure all templates use the unified `asset` or `webpack` helpers.
- **Global Availability**: Ensure libraries like jQuery are properly exposed if necessary via `ProvidePlugin` or explicit window assignment in entries, although IIFE output usually mitigates the strict need compared to ESM.

## Verification Plan

### Automated Tests

- **Unit Tests**: Create `t/99-webpack.t` to verify the `Webpack` plugin (manifest parsing, dev server detection).
- **Integration Tests**: Run `t/ui/*.t` to ensure the UI still renders correctly and all JS/CSS functionality is preserved.
- **CI**: Ensure `make generate-assets` passes in the CI environment.

### Manual Verification

1.  Run `npm run dev` and verify that the openQA UI works with HMR/live reloading.
2.  Run `npm run build` and check the contents of `public/dist/`.
3.  Run openQA in `production` mode and verify that assets are loaded from hashed paths.

## Rollback Plan

- Revert changes to `lib/OpenQA/Assets.pm`.
- Restore `assetpack.def` and `assetpack.yml`.
- Revert `Makefile` and `package.json`.
